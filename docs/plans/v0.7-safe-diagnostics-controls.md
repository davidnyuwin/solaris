# Solaris v0.7: Safe Diagnostics Controls Plan

This plan defines the architectural roadmap, candidate features, safety constraints, and implementation batches for the **Solaris v0.7: Safe Diagnostics Controls** milestone.

---

## Status

**Complete:** Batch 1 (Manual refresh, timestamp, loading state, error handling, accessibility), Batch 2 (Log pause/resume, copy redacted summary), Batch 3 (Auto-refresh scheduler, persistent Privacy Mode), and Accessibility Polish Pass — all implemented in `v0.7.0-dev`.

---

## Goals

The core objective of v0.7 is to enrich user control, interaction capabilities, and accessibility features over the Local Diagnostics dashboard, without violating sandbox boundaries or introducing network dependencies:
*   **Manual Control:** Empower users with active controls to trigger diagnostic refreshes and toggle logging ingestion streams on demand.
*   **Privacy & Exporting:** Provide safe mechanisms to copy redacted diagnostics parameters to the system clipboard for developer troubleshooting.
*   **Availability Clarity:** Enrich process status rows with more descriptive, actionable CLI availability and timeout diagnostics.
*   **Accessibility Integration:** Incorporate standard macOS voiceover accessibility markers across diagnostic panels.

---

## Non-Goals

*   **No Configuration/Profile Writing:** Solaris will not set, modify, or rewrite any parameters inside `~/.hermes/config.yaml`.
*   **No Configuration/Profile Reading:** Deferring any direct YAML or profile catalog parsing.
*   **No Command Steering:** The app will not execute mutating commands like restarting gateways or modifying active inference hosts.
*   **No WebSocket / Network Channels:** No WebSocket or network-based daemon control loops will be introduced.

---

## Candidate Features

1.  **Manual Refresh Button:** An explicit refresh control inside the Local Diagnostics title block triggering an asynchronous diagnostics sweep.
2.  **Keyboard Shortcut for Refresh:** Mapping standard `Cmd + R` to trigger diagnostics sweeps instantly.
3.  **Last Checked Timestamp:** A visual label indicating the exact hour/minute/second of the last successful diagnostics sweep.
4.  **Refresh Interval Selector:** An interactive picker/dropdown allowing developers to select their background diagnostics sweep frequency (e.g., Manual only, 5s, 10s, 30s).
5.  **Pause/Resume Log Updates:** A play/pause toggle in the logs rail allowing users to freeze the monospaced diagnostics log console during live tailing.
6.  **Copy Redacted Diagnostics Summary:** A dedicated button to export a redacted plain-text diagnostics snapshot to the clipboard, scrubbing all absolute paths and local developer usernames.
7.  **Privacy Mode Default-On Option:** A persistence toggle inside Settings allowing developers to force Privacy Mode active on app launch.
8.  **Clearer CLI Unavailable/Timeout UI:** Visually styling the Hermes CLI Status row depending on failure states (e.g. Warning badges for timeouts, Rose badges for missing bundle Python).
9.  **Accessibility (a11y) Labels:** Adding descriptive voiceover accessibility attributes (`.accessibilityLabel`, `.accessibilityHint`, `.accessibilityValue`) on status cards, checklists, and log outputs.

---

## Safety Constraints

*   **No Shell Evaluation:** All manual or periodic sweeps must continue to invoke native `Process` objects with rigid allowlisted argument arrays (e.g. `hermes status`).
*   **Continuous Path Redaction:** The clip-board export utility must reuse the verified `HermesCLIParsers.redactPath` static scanner on all exported text buffers.
*   **Non-Blocking UI Threading:** Periodic timer loops and refresh button triggers must execute asynchronously using Swift structured concurrency on a detached background thread to prevent GUI lockups.
