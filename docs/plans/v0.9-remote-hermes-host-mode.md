# Solaris v0.9 — Remote Hermes Host Mode

## Product Correction

Solaris is being developed on a MacBook Pro, but **Hermes Agent runs on a
separate Hermes host**.  Solaris must not assume Hermes data, sessions, logs,
config, or CLI are local to the MacBook.

The v0.1–v0.8 milestones built valuable native macOS UI patterns, local
diagnostics helpers, and safe control surfaces — but they assumed a local Hermes
installation.  That assumption is incorrect for the primary use case.

Solaris should become a native macOS companion for a **remote Hermes host**,
accessed primarily over SSH.

---

## Current Wrong Assumption

The current codebase assumes the MacBook runs Hermes Agent:

* `LocalHermesDiagnosticsService` reads local processes (`pgrep`, `lsof`),
  local log files (`~/.hermes/logs/agent.log`, `~/.hermes/logs/gateway.log`),
  and runs `hermes` CLI from the bundled Hermes Studio.app.
* `LiveHermesService` connects to `localhost:9119`.
* Settings UI shows mode switches that assume local transport.

This is useful for developers running Hermes locally but is not the product's
target deployment.

---

## New Source-of-Truth Model

```
Solaris.app on MacBook
  │
  ├── SSH connection layer ────── Remote Hermes host (source of truth)
  │
  ├── Local Diagnostics Mode  ─── MacBook local state only (dev/support)
  │
  └── Mock Mode  ──────────────── Demos and UI development
```

* **Remote Hermes host is the source of truth.**
* Solaris connects over SSH.
* All read-only checks run on the remote host.
* All chat/session operations run on the remote host.
* Local filesystem state on the MacBook is never treated as authoritative.

## Application Modes (Future)

### Mock Mode
* Existing in-memory mock service.
* For demos, screenshots, UI development.
* No Hermes host required.

### Local Diagnostics Mode
* Diagnostics for the MacBook's own state when Hermes runs locally.
* Not the primary mode.

### Remote Host Mode (NEW — v0.9 target)
* Primary real mode.
* Connects to a remote Hermes host over SSH.
* Runs allowlisted read-only commands via SSH.
* Plans to support remote `hermes chat` sessions via SSH PTY.

### Experimental Local REST Mode
* Optional. Only works when `hermes dashboard` is running locally or
  when an SSH tunnel is explicitly configured.

## Security Constraints

1. **No SSH keys or passwords stored in Solaris.**
   Rely on the macOS SSH agent (`ssh-agent`), Keychain, and `~/.ssh/config`.
2. **No command outside the allowlist.**
   Every remote command must be enumerated; no free-form SSH shell.
3. **No remote config file reading.**
   No `cat ~/.hermes/config.yaml`, no `cat ~/.hermes/.env`.
4. **No external platform messaging.**
   `hermes send` is blocked on the allowlist.
5. **No gateway lifecycle commands.**
   `hermes gateway restart/stop/start` are blocked.
6. **No model/provider mutation.**
   `hermes model set` is blocked.
7. **Output redaction before display.**
   Remote command output must be parsed or redacted.

## Safe Remote Command Allowlist (Batch 1)

```
hermes --version                    # Verify Hermes is installed on remote
hermes status                       # Read-only component status
hermes gateway status               # Read-only gateway status
which hermes                        # Locate Hermes binary
pwd                                 # Verify remote working directory
```

**Explicitly blocked:**

```
hermes send                         # External platform messaging
hermes model                        # Model/provider selection (mutating)
hermes model set                    # Model assignment (mutating)
hermes gateway restart              # Service lifecycle
hermes gateway stop                 # Service lifecycle
hermes config                       # Config viewing (may leak secrets)
cat ~/.hermes/config.yaml           # Raw config file access
cat ~/.hermes/.env                  # Raw secrets file access
```

## Chat Transport Options (Batches 2+)

### Option A: SSH PTY running `hermes`
Closest to real Hermes workflow.  Requires a PTY bridge library or custom
process wrapper.  No web UI required.  Uses host source of truth.

**Pros:**
* Full interactive `hermes` session.
* No dashboard server needed.
* Real Hermes environment.

**Cons:**
* Terminal emulation complexity (ANSI parsing, resize events).
* Needs PTY library integration.

### Option B: SSH one-shot `hermes chat -q`
Simpler than a full PTY.  Send prompt, get response, exit.

**Pros:**
* Easy to implement with `ssh user@host hermes chat -q "prompt"`.
* No terminal emulation.
* Fast to probe.

**Cons:**
* Session lifecycle per-command.
* Limited interactivity.
* Response may contain ANSI or tool call artifacts.

### Option C: SSH tunnel to remote dashboard
Forward remote port 9119 to local port and use the dashboard API.

**Pros:**
* Can reuse dashboard API.
* Standard tunnel.

**Cons:**
* Requires dashboard running on remote host.
* Token auth complexity.
* Not the core transport.

### Option D: Custom remote Solaris bridge
A small daemon on the remote host that exposes a controlled API.

**Pros:**
* Controlled protocol.
* Can enforce allowlist server-side.

**Cons:**
* Extra daemon to maintain.
* Additional trust surface.
* Unnecessary until SSH paths are exhausted.

## Non-Goals (v0.9)

* No WebSocket client implementation.
* No `hermes send` integration.
* No local config file reading.
* No storing SSH keys or passwords in Solaris.
* No installing software on the remote Hermes host.
* No mutating Hermes commands.
* No chat UI yet (beyond probe/verification).
