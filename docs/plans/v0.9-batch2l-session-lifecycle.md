# Solaris v0.9 — Batch 2L: Session Lifecycle Management

This document establishes the plan, implementation details, safety rules, and UX flows for managing chat session lifecycles in Solaris v0.9 Batch 2L.

---

## 1. Safe Session Lifecycle Operations

Solaris introduces three new user-facing actions to manage chat session states:

1. **Rename Session:** Allows users to manually set custom titles for their chat sessions.
2. **Delete Session:** Permits the permanent removal of individual sessions from the local history.
3. **Clear All History:** Destructive action to purge all local chat sessions, resetting to a clean default state.

---

## 2. Active Stream Guard

To ensure consistency and prevent race conditions or database corruption:
* All destructive lifecycle actions (Rename, Delete, and Clear All) are blocked if a chat stream is active (`chatState.isActive` is true).
* If a user attempts to rename, delete, or clear all sessions during active execution, Solaris blocks the operation, prevents showing any confirmation dialogs, and displays a user-visible warning banner:
  > Finish or cancel the active chat before managing sessions.
* Active streams are never cancelled automatically for lifecycle actions; the user must explicitly let the stream complete or click Cancel first.

---

## 3. Title Sanitisation & Privacy Rules

To protect developer privacy and prevent credential leakage through auto-generated or manually typed titles:
* **Characters & Whitespace:** Leading and trailing whitespaces are trimmed, and all control characters are stripped.
* **Length Capping:** Titles are strictly capped using the project-wide 40-character limit. Titles exceeding this cap are truncated, and `...` is appended.
* **Empty Fallback:** Empty titles are rejected and automatically fall back to `New Chat`.
* **Path & Token Redaction:** Titles flow through `DiagnosticsRedactor.redact(_:)`, which automatically:
  * Replaces `/Users/<username>` directory paths with `~`.
  * Redacts secret keys, API tokens (such as OpenAI keys), and credentials.
* **Manual Override Integrity:** When a session is manually renamed by a user, a new backward-compatible optional flag `isManuallyRenamed` is set to `true` on the session model. This prevents the auto-title generator from overwriting the manual title during future chat turns in that session.

---

## 4. Destructive Action & Confirmation Flows

To prevent accidental data loss, destructive actions require explicit user confirmation:

### Delete Session
* **User Flow:** Triggered via context menu on a session list row or clicking the ellipsis Menu button.
* **Destructive Boundary:** Deleting a session completely removes it and all associated runs from local memory and persists the change to the local JSON file.
* **Fallback Safety:**
  * If the active session is deleted, Solaris automatically switches focus to the most recently updated remaining session.
  * If the deleted session was the *only* session, Solaris immediately initializes a clean fallback empty session (`New Chat`) to ensure the application state remains valid and no orphaned runs are left in the UI.

### Clear All History
* **User Flow:** Triggered via a trash icon next to the "New Chat" button in the Chat History header.
* **Destructive Boundary:** Purges all chat sessions and runs. Resets the document to a clean schema version 1 state with a single empty fallback session (`New Chat`).
* **Explicit Warning Copy:**
  > Clear all local chat history? This removes saved Solaris chat sessions from this Mac only. It does not affect Hermes, SSH settings, or remote data.
* **Settings & Remote Protections:**
  * Clear All only affects `chat-history.json`.
  * It **does not** touch remote Hermes data, developer gate settings, local SSH configs, or local settings (e.g. Remote Host, User, Port).
  * Corrupt JSON backups (e.g. `chat-history.corrupt.<timestamp>.json`) are left untouched.

---

## 5. UI Layout

The session sidebar (`RunsView`) has been updated to integrate these features cleanly using the existing dark/purple/graphite aesthetics:
* **Sidebar Controls:** Each session row includes an ellipsis button showing a Menu with "Rename" and "Delete" actions. A context menu is also bound to the row for right-click accessibility.
* **Gesture-based Selection:** Session rows use `.onTapGesture` rather than nested `Button` elements to eliminate event propagation conflicts when clicking the ellipsis menu.
* **Header Actions:** A new trash button is placed in the "Chat History" header next to the "New Chat" button to trigger the Clear All flow.

---

## 6. Remaining Blockers

* **Cloud Sync:** Solaris remains fully local-first. No cloud storage synchronization is planned.
* **Encryption at Rest:** Local JSON files are saved in the user's Application Support directory and are not encrypted by the app.
