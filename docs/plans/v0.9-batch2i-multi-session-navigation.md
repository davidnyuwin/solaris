# Solaris v0.9 — Batch 2I: Multi-Session Chat Navigation UI

This document details the multi-session management, navigation UI, title generation rules, and active-stream switching guards implemented in Solaris v0.9 Batch 2I.

---

## 1. Session Navigation UI

A new two-pane layout is introduced inside the `RunsView` (Recent Runs panel):

```
┌───────────────────────────────────────┬───────────────────────────────────────┐
│              Chat History             │                Recent Runs            │
├───────────────────────────────────────┼───────────────────────────────────────┤
│  [+] New Chat                         │  [Filter runs...]                     │
│                                       │                                       │
│  • Deploy Database              (2)   │  ┌─────────────────────────────────┐  │
│    config for ~/test with token...    │  │ Prompt: deploy database         │  │
│    12:45 PM                           │  │ Response: Succeeded.            │  │
│                                       │  └─────────────────────────────────┘  │
│  • New Chat                     (0)   │  ┌─────────────────────────────────┐  │
│    12:40 PM                           │  │ Prompt: show gateway status     │  │
│                                       │  │ Response: Active.               │  │
│                                       │  └─────────────────────────────────┘  │
└───────────────────────────────────────┴───────────────────────────────────────┘
```

*   **Left Column (Session Sidebar):**
    *   **New Session Button:** A header button `[+]` allows creating a new chat session.
    *   **Scrollable List:** Shows all persisted sessions sorted by the `updatedAt` timestamp (newest first).
    *   **Session Metadata:** Each row displays the session title, last-updated time, run count bubble, and a single-line preview of the last command prompt.
    *   **Selected Session Visual State:** The selected session is highlighted with a background overlay and bordered with the `hermesPurple` brand theme color.
*   **Right Column (Command Runs):**
    *   Displays the runs associated with the selected active session.
    *   Exposes a real-time validation banner at the top if session switching is blocked.

## 2. Active Session Model & State

The view model (`HermesViewModel`) manages multi-session state through:

*   `activeSessionID: UUID?` - The identifier of the selected session.
*   `sessions: [HermesChatSession]` - Computed property exposing all available sessions in history.
*   `activeSession: HermesChatSession?` - Computed property retrieving the details of the active session.
*   `runsForActiveSession: [HermesRun]` - Computed property resolving and sorting the runs of the current session.

New chat runs initiated from the Dashboard always attach to the active session.

---

## 3. Session Switching Guards

To prevent data corruption, in-progress stream loss, or orphaned child processes:

*   **Active Stream Switching Rule:** If the chat state is active (`chatState.isActive` evaluates to true, i.e. `.connecting` or `.streaming`), switching sessions or creating a new session is blocked.
*   **User Feedback:** The UI displays an error message banner:
    `Finish or cancel the active chat before switching sessions.`
*   **No Silent Cancellation:** The active SSH process stream is never implicitly aborted or changed.

---

## 4. Title Generation and Privacy

To preserve privacy and maintain a clean layout, session titles are derived automatically:

*   **Default Title:** A new session starts with `New Chat`.
*   **Derivation Trigger:** After the first terminal run completes in a session, the title is generated from the sanitised prompt preview.
*   **Command Sanitisation:** Any `/chat ` or `chat ` command prefixes are stripped.
*   **Length Cap:** Titles are capped at exactly **40 characters**. Longer titles are truncated and appended with `...`.
*   **Privacy Filtering:** All titles are derived from `promptPreview`, which is pre-sanitised by `OutputSanitiser.sanitise(_:isStreaming:)`. Therefore, session titles never contain:
    *   API keys or bearer tokens (redacted as `Bearer [REDACTED]` or `[REDACTED]`).
    *   Absolute paths like `/Users/username/` (normalised to `~/`).
    *   Non-printable ASCII control characters.

---

## 5. Persistence Interaction

*   **Auto-Save:** Any session modifications (such as appending runs, updating timestamps, or changing titles) automatically serialize and trigger atomic background writes through `ChatHistoryStore`.
*   **Empty State Recovery:** If the loaded history file contains no sessions, `loadChatHistory()` creates a default `"New Chat"` session automatically.
*   **Diagnostics commands integration:** When a system diagnostics or status check is executed, the fetched background command runs are preserved and merged alongside the active chat session runs in the UI list.

---

## 6. Deferred Features

*   **Markdown Rendering:** Chat cards continue to render raw plain-text output. Visually styled markdown bubbles are deferred.
*   **Polished Chat Bubbles:** The dashboard card design remains consistent; styled bubbles are deferred to a subsequent frontend design pass.
*   **Session Deletion/Renaming:** Interactive manual title editing or session deletion is deferred to a future batch.
