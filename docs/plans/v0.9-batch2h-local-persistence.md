# Solaris v0.9 — Batch 2H: Local Chat History Persistence

This document details the lightweight, local JSON-based persistence design implemented in Solaris v0.9 Batch 2H to preserve completed chat sessions and run cards across application restarts.

---

## 1. Persistence Architecture

Solaris implements a simple, inspectable, and secure local file-based storage architecture. The store decouples the complex in-memory SwiftUI/Combine UI-bound models from the JSON schema layout using a specialized mapping layer.

```
┌─────────────────────────────────┐
│     HermesViewModel (UI)        │
└────────────────┬────────────────┘
                 │
  (Terminal state reached:       
   completed, failed, timedOut,   
   or cancelled)                 
                 │
                 ▼
┌─────────────────────────────────┐
│       OutputSanitiser           │ (Enforces UTF-8, ANSI stripping,
└────────────────┬────────────────┘  and API key/path redactions)
                 │
                 ▼
┌─────────────────────────────────┐
│    HermesPersistedRun (Model)   │ (Maps to UUID, Dates, capped prompts,
└────────────────┬────────────────┘  and cleaned responses)
                 │
                 ▼
┌─────────────────────────────────┐
│      ChatHistoryStore           │ (Thread-safe background queue writes,
└────────────────┬────────────────┘  atomic file replacement)
                 │
                 ▼
      [ chat-history.json ]
```

---

## 2. Storage Location

Persistence uses the standard system application support folder:

*   **Production path:**
    `~/Library/Application Support/Solaris/chat-history.json`
*   **Test Isolation:**
    Tests construct a temporary directory path dynamically using `FileManager.default.temporaryDirectory` appended with a unique UUID. Tests never read or write to the user's real Application Support directory.

---

## 3. Data Models and Schema Versioning

To ensure safe future upgrades, the file contains a root document version wrapper:

```swift
struct HermesChatHistoryDocument: Codable {
    var schemaVersion: Int
    var sessions: [HermesChatSession]
}

struct HermesChatSession: Codable, Identifiable, Equatable {
    var id: UUID
    var createdAt: Date
    var updatedAt: Date
    var title: String
    var runs: [HermesPersistedRun]
}

struct HermesPersistedRun: Codable, Identifiable, Equatable {
    var id: UUID
    var createdAt: Date
    var completedAt: Date?
    var mode: String
    var promptPreview: String?
    var response: String
    var status: String
    var errorSummary: String?
}
```

*   **Current Schema Version:** `1`
*   **Versioning Policy:** Increments to `schemaVersion` will require a custom migration step inside `ChatHistoryStore.load()`. If migration is not possible, the store will back up the old version and load a fresh document.

## 4. Storage Behavior

### Missing File
If `chat-history.json` does not exist at startup, the store returns a blank `HermesChatHistoryDocument(schemaVersion: 1, sessions: [])` and continues normally. No error is thrown.

### Corrupt File Handling
To prevent data loss:
1. If loading fails due to malformed or corrupted JSON, the store does not crash.
2. The store automatically renames the corrupt file to `chat-history.corrupt.<timestamp>.json` within the same folder.
3. A blank document is returned, allowing the user to start a new clean history without losing the corrupt file (which remains inspectable on disk).

### Atomic Writes
Writes to the file system are atomic:
1. The JSON data is serialized and written to a temporary file (`chat-history.tmp`) in the same directory.
2. If successful, `FileManager.default.replaceItemAt` is used to swap the old file with the new one.
3. This guarantees that if the application crashes mid-save, the existing history file remains intact and uncorrupted.

---

## 5. ViewModel Integration

### Startup Load
At startup, `loadAllData()` triggers `loadChatHistory()`, pulling the persisted history into memory. If no sessions exist, a default active session is created.

### Save Triggers
To avoid saving incomplete or temporary data:
*   Only terminal runs are persisted: `.completed`, `.failed(String)`, `.timedOut`, or `.cancelled`.
*   Transient active streaming states (e.g. `.validating`, `.connecting`, `.streaming`) are **never** saved.
*   Autosave triggers automatically when a run finishes or is cancelled/timed out.
*   Duplicate checks on `runs` are enforced via the run's stable `id` (mapped to a stable UUID) to prevent duplicate runs from appending.

## 6. Privacy and Sanitisation Rules

Persistence complies with the security guidelines established in the threat model:

1.  **No Raw Prompts:** Full prompt text is never saved to the history document. Only a short preview, sanitised and capped to **120 characters**, is stored.
2.  **Explicit Output Sanitisation:** Every saved response runs through `OutputSanitiser.sanitise(_:isStreaming:false)` before being written to disk, ensuring that ANSI colors, OSC hyperlink controls, system paths (`/Users/sysadmin/`), and API credentials (`sk-...`) are stripped/redacted.
3.  **Strict Output Limits:** Stored response text is capped at **64KB**. Truncated responses include an explicit indicator (`... [Output truncated to 64KB for safety]`).
4.  **No Developer Credentials:** SSH hostnames, passwords, remote command paths, and usernames are excluded from persisted run entries.
5.  **No Raw Process Logs:** Raw stdout/stderr pipelines are excluded. Only the sanitised, display-safe final message is persisted.

## 7. Remaining Production Blockers

*   **Multi-Session Navigation UI:** The sidebar navigation for switching between multiple historical chat sessions is not implemented.
*   **Markdown Chat Bubbles:** Responses are saved as plain text; a styled markdown presentation layer is required for a production-grade experience.
*   **Encryption at Rest:** Persisted history is currently plaintext JSON. Production should encrypt this data via the Keychain or Secure Enclosure.
