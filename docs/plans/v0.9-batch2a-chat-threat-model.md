# Solaris v0.9 — Batch 2A: Remote Chat Threat Model & Sanitisation Design

This design document establishes the security guidelines, output sanitisation pipeline, command execution boundaries, and failure policies for integrating remote `hermes chat -q` support into Solaris.

> [!IMPORTANT]
> **Implementation Blocked:** No code changes implementing live chat execution, prompt routing, or streaming chat UI are to be introduced. Batch 2 implementation remains explicitly blocked until this design is reviewed and approved.

---

## 1. Trust Boundaries & Threat Model

```
 ┌────────────────────────┐
 │   Solaris Companion    │  (Trusted Local Environment)
 └───────────┬────────────┘
             │
   SSH Transport Tunnel      (Encrypted, Authentication via ssh-agent)
             │
 ┌───────────▼────────────┐
 │  Remote Hermes Host    │  (Executes allowed binary, but outputs are UNTRUSTED)
 └────────────────────────┘
```

Solaris runs in the local macOS user session. All remote commands execute on a separate Hermes host via `/usr/bin/ssh`.

### Trust Boundaries
1. **Local Companion (Solaris.app):** Trusted. Handles user input, configurations, and sanitisation.
2. **SSH Transport Channel:** Encrypted transit. Handled natively by `/usr/bin/ssh` and macOS `ssh-agent`. No private keys or passwords are store by Solaris.
3. **Remote Hermes Host & CLI:** Execution is trusted for standard inputs, but the **raw console outputs** (consisting of stderr, stdout, and LLM text generation) must be treated as **Untrusted**.
4. **Model Output (LLM Generation):** Untrusted. Model outputs can contain arbitrary user/attacker-influenced text, rendering it susceptible to prompt injection, system path leaks, secret key leaks, and terminal control sequence injections.

### Identified Risks & Mitigations

| Threat | Description | Mitigation Strategy |
| :--- | :--- | :--- |
| **Command Injection** | Attackers injecting shell metacharacters (e.g. `; rm -rf /`) into the chat prompt to execute arbitrary code on the remote host. | **Remote-Shell Aware Allowlist & Parameter Isolation:** Even when Solaris launches `/usr/bin/ssh` via local `Process` argument arrays, OpenSSH reconstructs a single command string on the remote host and executes it through the user's default login shell. Therefore, *never* interpolate raw user prompts or dynamic variables directly into the remote command string. In this batch, chat execution is blocked entirely. |
| **Terminal Escape Injection** | Remote stdout returning ANSI, OSC (Operating System Command), or cursor-control escape sequences that hijack the local terminal host or spoof the application UI. | **ANSI/OSC Stripping:** Run a strict regular expression filter to strip all ANSI colors, text styling, cursor movements, and OSC sequences (such as OSC 8 hyperlinks and OSC 52 clipboard writes). |

| **Secret & Key Leakage** | The remote CLI output or LLM traceback output containing API keys, Bearer tokens, or passwords. | **DiagnosticsRedactor Integration:** Apply strict regular expression matching to redact OpenAI keys (`sk-...`), Bearer tokens, and hexadecimal hashes before rendering to the UI. |
| **System Path Leakage** | Diagnostic messages revealing absolute system paths (e.g., `/Users/sysadmin/...`) containing sensitive usernames or local config paths. | **User Path Substitution:** Replace all instances of the home directory path prefix (`/Users/[a-zA-Z0-9_-]+`) with `~` in the output display. |
| **Denial of Service (DoS)** | The remote LLM generation hanging indefinitely, blocking thread execution or consuming excessive host resources. | **Strict Timeout & Process Termination:** Enforce a hard 30-second timeout. Terminate the SSH subprocess immediately using `Process.terminate()` if the timeout fires or the user cancels the query. |
| **UI Spoofing** | The model returning markdown structures or status headers that visually mimic real system error messages or authentication prompts. | **Scope-Isolated UI Rendering:** Display all chat responses inside clearly demarcated, container-restricted chat message bubbles. Never parse or render model output as system-level alerts. |

## 2. Strict Command Execution Boundary

To prevent command injection, the application must adhere to a strict argument boundary.

### Rules
1. **No Shell Invocations Locally:** Solaris must call `/usr/bin/ssh` directly as the process executable. It must never invoke a local shell (like `/bin/sh` or `/bin/zsh`) to evaluate SSH command strings.
2. **Remote Shell Interpretation Awareness:** Although Solaris passes arguments as an array locally, OpenSSH reconstructs them into a single string to execute via the user's default remote login shell (e.g. `/bin/bash -c "..."`).
   * **Target Command Structure:**
     `["-p", port, "-o", "BatchMode=yes", "-o", "ConnectTimeout=5", "-o", "StrictHostKeyChecking=accept-new", userAtHost, hermesCommandPath, "chat", "-q", prompt]`
   * **Critical Restriction:** Because OpenSSH joins arguments and runs them via a remote shell, passing raw prompt text inside command-line arguments is highly unsafe.
   * **Chat Execution Status:** Chat execution remains **blocked / stubbed** in this batch. If prompt routing is implemented in the future, it must use a transport mechanism that avoids remote shell expansion of user text (such as passing prompts via `stdin` or a dedicated bridge process).
3. **Hermes Path Sanitisation:** The custom `hermesCommandPath` must be parsed using `sanitiseHermesCommand` to reject whitespace, semicolons, pipes, or command separators.

## 3. Strict Output Sanitisation Pipeline

Every line of output returned from the remote SSH process must flow through the following sequentially ordered pipeline before being rendered to the UI or written to logs:

```
                  ┌──────────────────────┐
                  │  Raw Remote Output   │
                  └──────────┬───────────┘
                             │
                      1. Size Limiting   (Reject if > 64KB)
                             │
                    2. UTF-8 Validation  (Replace invalid bytes with \u{FFFD})
                             │
                 3. Control Char Stripping  (Strip 0x00-0x08, 0x0B-0x0C, 0x0E-0x1F, 0x7F)
                             │
                     4. ANSI/OSC Removal  (Strip escape and OSC sequences)
                             │
                      5. Redaction Step  (Redact keys, paths, and PIDs)
                             │
                  ┌──────────▼───────────┐
                  │ Display-Safe Output  │
                  └──────────────────────┘
```

### Pipeline Rules
1. **Size Limiting:** Stop reading and discard output if the cumulative size exceeds **64KB (65,536 bytes)**. This prevents memory bloat from massive model responses or infinite loops.
2. **UTF-8 Validation:** Enforce valid UTF-8 encoding. Invalid byte sequences must be replaced with the standard unicode replacement character `\u{FFFD}` rather than throwing a parsing error.
3. **Control Character Stripping:** Remove all non-printable ASCII control characters. Preserve only standard formatting whitespaces (`\n`, `\r`, `\t`). Strip characters in the ranges `0x00...0x08`, `0x0B...0x0C`, `0x0E...0x1F`, and `0x7F`.
4. **ANSI/OSC Removal:**
   * **ANSI Escape Sequences:** Match and remove `\u{001B}\[[0-9;]*[a-zA-Z]` to strip ANSI colors and text styles.
   * **OSC Sequences:** Match and remove Operating System Command patterns:
     `\u{001B}][0-9];.*?\u{0007}` and `\u{001B}][0-9];.*?\u{001B}\\`. This is critical to prevent malicious text from writing to the system clipboard or opening links.
5. **Redaction & Substitution:**
   * **API Keys:** Redact patterns matching `sk-[A-Za-z0-9]{20,}` or `bearer[[:space:]]+[A-Za-z0-9_\-\.]{12,}`.
   * **User Directories:** Identify absolute path prefixes `/Users/[a-zA-Z0-9_-]+/` and substitute them with `~/`.
   * **PIDs:** Detect status-card PIDs using `PID(s): \d+` or `Process ID: \d+` and mask them.
6. **Logging Policies:**
   * **Prompt Logs:** The text of the user's prompt must never be written to application logs.
   * **Response Logs:** The text of the model's response must never be written to application logs.
   * **Log Metadata:** Only log connection details (duration, remote hostname, exit code, stdout/stderr byte sizes, and error/success states).

## 4. Timeout, Cancellation, and Failure Policy

### Process Lifecycle
* **Active Timeout:** The execution limit for a chat command is set to **30 seconds** (giving the LLM sufficient time to generate answers).
* **Cancellation Signal:** If the user clicks "Cancel" or closes the view during execution, the task must call `.cancel()` on the structured concurrency task, which in turn calls `process.terminate()`.

### Failure Matrix

| Failure Mode | Detection Criteria | User-Facing Message | UI Presentation |
| :--- | :--- | :--- | :--- |
| **Command Timeout** | Process active after 30 seconds; terminated by local watch task. | "The connection timed out after 30 seconds. Verify remote host performance." | Error Banner (Orange) |
| **SSH Auth Failure** | Process exits with status code `255` and stdout is empty. | "SSH authentication failed. Verify settings or ensure your ssh-agent contains the required key." | Critical Error State (Red) |
| **Hermes Missing** | Process exits with status code `127` or `1` during version query. | "Hermes CLI binary not found on the remote host at the configured path." | Configuration Setup State |
| **Command Not Found** | CLI returns status code indicating sub-command invalidation. | "The 'chat' subcommand is not supported on this version of Hermes." | Diagnostic Alert Banner |
| **Sanitisation Failure** | Output is empty or rejected during size/encoding validation. | "Safety check: The remote output was blocked because it contained unsafe sequences." | Critical Alert State |
| **Host Unreachable** | SSH exits with status indicating connection refused or route error. | "Unable to connect to the remote host. Verify network connection and port settings." | Connection Failure State |

## 5. Acceptance Criteria

Before Batch 2 implementation is marked complete, it must pass the following checks:
1. **Safety Assertions:**
   - [ ] No raw shell commands can be constructed using user inputs.
   - [ ] ANSI and OSC sequences are fully stripped from mock/live streams.
   - [ ] Absolute developer/system paths (`/Users/sysadmin`) are redacted and shown as `~`.
2. **Lifecycle Controls:**
   - [ ] The chat execution process successfully aborts remote execution when the local timeout fires.
   - [ ] User cancellation terminates the SSH process immediately on disk.
3. **Log Sanitisation:**
   - [ ] No user-submitted prompt text is printed to local file system logs.
   - [ ] No generated model response text is printed to local file system logs.
4. **UI Decoupling:**
   - [ ] The chat input panel is disabled while a query is running.
   - [ ] A clean loading spinner is shown during generation.

## 6. Batch 2C Safe Prompt Transport Discovery & Findings

- **Introspection Command Verified:** `ssh macmini-cf "hermes chat --help"`
- **Prompt Transport Strategy Discovery:**
  - *Standard Input (stdin) support:* **CONFIRMED**. Running `echo "help" | hermes chat -q - -Q` on the remote host successfully reads the query from standard input (indicated by the `-` value passed to the query parameter), processes a single turn, and outputs the response to stdout (exiting with code 0).
  - *Mitigation against command injection:* This fully removes the requirement to interpolate raw user prompt text into the SSH remote command string arguments. The SSH arguments list remains completely static.
- **Future Safe Transport Architecture (Batch 2D+):**
  ```
  Local Prompt Text
        │
        ▼
  Local Validation (Check bounds and formats)
        │
        ▼
  SSH Process Standard Input (stdin) Pipe
        │
        ▼
  Fixed SSH remote command: hermes chat -q - -Q (Standard args, no user strings)
        │
        ▼
  Remote Shell execution (Fixed command string, safe from injection)
        │
        ▼

        │
        ▼

        │
        ▼

  ```

## 7. Batch 2D Stdin Executor Implementation & Safety Controls

Batch 2D successfully implemented safe standard input (stdin) piping for `RemoteSSHExecutor` without exposing prompt text to remote shell command-line arguments.

- **API Signature Added:**
  ```swift
  public func execute(
      command: RemoteHermesCommand,
      settings: RemoteHostSettings,
      timeout: TimeInterval = 8,
      stdinData: Data? = nil
  ) async -> RemoteSSHResult
  ```
- **Stdin Input Size Limit:** Enforced at a strict **16KB (16,384 bytes)** check performed before process spawn. If `stdinData` exceeds 16KB, execution is blocked immediately and a failure result is returned without launching the process.
- **Pipe Lifecycle Security:**
  - Standard input is configured with a dedicated `Pipe` if `stdinData` is provided.
  - The process is spawned, stdin bytes are written to the pipe, and the write handle is closed immediately via `close()` to signal EOF (preventing the remote process from hanging indefinitely).
  - The write handle is closed reliably under all circumstances (success, write failure, cancellation, timeout).
  - The subprocess is terminated immediately if the stdin write fails, preventing orphaned commands.
- **Chat Commands Blocked:** Live remote `/chat` execution remains blocked via `RemoteCommandBuilder.buildArguments` returning `.chatExecutionNotYetApproved`.
- **Zero Leakage / Logging Policy:** Neither the stdin prompt nor the raw stdout/stderr is printed or logged. Only execution metadata (duration, success/failure state, byte counts, and exit code) is recorded.
- **Verification Status:**
  - Build and tests compiled.
  - Stdin data flow and size limits validated via test suite with process-launching overrides.
  - Smoke test and secret scan passed.

## 8. Batch 2E Controlled Remote Chat Gate & Console Binding

Batch 2E successfully implemented a developer-controlled remote chat gate and integrated `/chat <prompt>` console input to execute against the remote Hermes host via SSH stdin.

- **Developer Gate Behavior:**
  - Guards chat execution using both a compile-time check (`#if DEBUG`) and a runtime setting (`UserDefaults.standard.bool(forKey: "EnableDeveloperRemoteChat")`).
  - **Fails closed:** By default, and in release builds, the gate is disabled.
  - When disabled: executing `/chat` displays the error: `Remote chat is disabled. Enable the developer remote chat gate to test stdin-based Hermes chat execution.`
  - When enabled: executing `/chat` logs a warning in the console: `Developer remote chat is enabled. Prompts are sent to the configured remote Hermes host via SSH stdin. Prompt text is not logged.`
  - Toggled via a UI switch in the "Developer Console" section of Settings.
- **Fixed Remote Command:**
  - Executes only: `[hermesCommandBase, "chat", "-q", "-", "-Q"]` (e.g. `/Users/sysadmin/.hermes/hermes-agent/.venv/bin/hermes chat -q - -Q`).
  - Prompt text is passed purely via standard input (`stdinData`) and never placed in SSH command arguments.
  - Arbitrary `.chat` command construction remains blocked.
- **Prompt Validation:**
  - Rejects empty prompts locally with `Chat prompt cannot be empty.`.
  - Rejects prompts larger than 16KB locally with `Chat prompt exceeds maximum allowed size of 16KB.`.
- **Output Sanitisation Path:**
  - Every response (both stdout and stderr) is filtered through `OutputSanitiser` before display.
  - Strips ANSI colors, cursor movements, OSC hyper-links, system paths, PIDs, and API keys.
  - Limits output to 64KB, appending a truncation marker.
- **Remaining Production Blockers:**
  - Stream-based response UI is not implemented; output is collected in full before displaying.
  - Streaming stdin/stdout via SSH PTY or WebSockets is currently out of scope.
  - Production gate validation and settings discovery are needed before enabling by default.

## 9. Batch 2F Stream-Safe Remote Output Pipe & Stateful Sanitisation

Batch 2F implemented stream-safe output piping for the remote Hermes SSH execution backend. This allows progress text to update the UI incrementally without exposing the companion app to split-chunk control sequence leaks or corrupt UTF-8 character boundaries.

### Stream Architecture

```
Raw SSH stdout/stderr bytes
        │
        ▼
StreamingOutputSanitiser (Buffers UTF-8 boundary bytes and parses ANSI/OSC states)
        │
        ▼
AsyncStream<RemoteSSHStreamEvent> (Yields typed events: .stdout, .stderr, etc.)
        │
        ▼
HermesViewModel (Accumulates raw text, runs global OutputSanitiser on final buffer)
        │
        ▼
SwiftUI View (Progressive run card updates)
```

### Event Model

```swift
public enum RemoteSSHStreamEvent: Sendable, Equatable {
    case stdout(String)
    case stderr(String)
    case status(String)
    case completed(exitCode: Int32)
    case failed(String)
    case timedOut
}
```

### Stream-Safe Sanitisation Strategy
To prevent escaping characters across chunk boundaries (such as a split ANSI CSI or OSC hyperlink sequence), the `StreamingOutputSanitiser` processes chunks statefully:
1. **UTF-8 Byte Split Buffer:** Checks the trailing bytes of incoming chunks. If a multi-byte UTF-8 character sequence is split across chunks (e.g. `[0xE3, 0x81]` of a 3-byte Hiragana character), the sanitiser buffers the partial bytes, and only decodes them when the remaining bytes arrive in the next chunk.
2. **Stateful Escape Parser:** Maintains state (`.normal`, `.sawEsc`, `.ansiCsi`, `.osc`, `.oscEsc`) across successive byte chunks. This guarantees that escape sequences split across chunk boundaries (e.g. `\u{001B}[3` followed by `1m`) are fully stripped and do not leak into the UI.
3. **Accumulated Safe Verification:** The view model accumulates raw streamed stdout/stderr and runs the complete `OutputSanitiser.sanitise` pipeline (verifying path substitution, regex-based secret redaction, and global size truncation) on the full buffer in real-time.

### stdout/stderr Handling
* **stdout:** Yielded as `.stdout(String)` containing incrementally sanitised text. Appended progressively to the current response text of the active run card.
* **stderr:** Yielded as `.stderr(String)` containing incrementally sanitised diagnostics. Appended progressively to the diagnostics log window, keeping developer logs separated from chat responses.

### Timeout & Cancel Behaviour
* **Timeout:** A 30-second watchdog task runs concurrently. If it fires while the stream is active, the executor calls `process.terminate()`, closes the stdin/stdout/stderr pipes, and yields a `.timedOut` event.
* **Cancellation:** If the user cancels execution, the Structured Concurrency task is cancelled. The stream's `onTermination` block immediately terminates the active SSH subprocess and closes the stdin write pipe.

### Developer Gate & Trust Constraints
* **Developer Gate remains active:** Developer remote chat is disabled by default, and compiled out of release builds via `#if DEBUG`. Toggled only in Settings.
* **Model Output is Untrusted:** Streamed model output is strictly text-only. The app must never parse model output to perform privilege escalation, mutate settings, or execute local shell scripts.
* **Remaining Blockers:**
  * Production chat UI design, message styling, and message bubble wrapping.
  * Local history persistence and session state management.
  * High-latency SSH connection recovery or automatic retry logic.
