# Solaris v0.10: Remote Runtime Reliability Planning

This document establishes the system design, proposed phases, safety parameters, and verification requirements for Solaris v0.10, focusing on Remote Runtime Reliability.

---

## 1. Goal
Ensure Solaris can reliably diagnose and connect to remote Hermes runtime environments (e.g. `KNOWHERE.local`), even when local configurations (SSH agent, passphrase, sandbox, or network routes) are unstable or misconfigured. Provide clear user-facing diagnostic guidance when connections fail.

## 2. Non-Goals
* Do not assume Hermes files or standard execution binaries exist on the local MacBook Pro build environment.
* Do not store keys, passwords, or secrets in local state or log files.
* Do not enable SSH agent forwarding.
* Do not modify or edit local user SSH configurations (`~/.ssh/config`).

## 3. User Problem
Developers run Solaris on their local Mac, but the Hermes agent runtime and data resides entirely on a remote host (e.g., `KNOWHERE.local`). When the remote connection fails (due to passphrase keys not being loaded in the local SSH agent, sandbox path access limits, or missing remote environment path directories), the app fails silently or prints general network errors. Users lack clear steps to diagnose if the issue lies in local agent setup, keys, sandbox boundaries, or remote binary installation.

## 4. Proposed Phases

### Phase 1: SSH Agent & Passphrase Preflight Diagnostics
* Add local check utilities to query the local `ssh-agent` state (`ssh-add -l`).
* Alert the user if no keys are loaded in the agent when they attempt connection without specifying an identity file.
* If a passphrase-protected identity file is selected but not added to `ssh-agent`, show clear instructions on how to load it (`ssh-add <path>`).

### Phase 2: Remote Hermes Environment Discovery
* Instead of assuming a static remote path or raw command, run remote probe commands to discover available python environments and script paths.
* Safe command construction using array argument separation (e.g., `["which", "hermes"]` or probing standard paths).

### Phase 3: Configurable Remote Working Directory
* Add a configuration field in Remote Host Settings for a remote working directory (e.g. `/Users/sysadmin/.hermes/hermes-agent`).
* Enforce metacharacter and space checks on the directory path, ensuring it is passed cleanly during remote execution.

### Phase 4: Extended Diagnostic Feedback & Sandboxing Guide
* Detect if sandbox limitations block access to local private key paths (e.g. `~/.ssh/`).
* Provide a troubleshooting interface explaining exactly why connection preflight failed (e.g., "SSH Agent lacks loaded keys", "Access to key file denied by App Sandbox", or "Remote path directory not found").

---

## 5. Security & Safety Constraints
* **Gated Execution:** Live remote chat operations remain strictly debug-gated and compile-gated behind `#if DEBUG`.
* **Argument Boundaries:** No shell-string concatenation for remote run commands. All commands must use direct Process array arguments.
* **No Secret Storage:** Passwords, keys, or passphrase texts are never requested or stored.

---

## 6. Files Likely Involved
* `Sources/HermesCompanion/Models/RemoteHostSettings.swift` (add remote working directory property and validation)
* `Sources/HermesCompanion/Services/RemoteSSHExecutor.swift` (update argument builder to handle remote directory switching and environment detection)
* `Sources/HermesCompanion/ViewModels/HermesViewModel.swift` (extended connection tests and preflight diagnostic logic)
* `Sources/HermesCompanion/Views/SettingsView.swift` (new settings field and diagnostic error guides)

---

## 7. Open Questions
* How can we reliably check local `ssh-agent` loaded keys list on macOS within the sandboxing sandbox limits if sandboxing is enabled in the future?
* Should Solaris automatically prompt to run a local terminal command via a helper script for `ssh-add` if no keys are active?
