# Solaris v0.11 — Remote Runtime Live Integration (Planning)

**Created:** 2026-06-04
**Status:** Planning — no implementation approved

## Overview

v0.10 established remote runtime readiness with mock/gated boundaries. v0.11 introduces live remote execution and remote Hermes discovery, following a strict architecture-first approach.

**Hard rule:** Do not combine live remote execution and remote Hermes discovery in the same implementation batch.

Sequence:
1. Architecture decision document.
2. Live execution safety contract.
3. Minimal live command probe.
4. Remote Hermes discovery.
5. UI integration.

---

## Phase 1: IPv6 Bracket Support

**Size:** Small, contained hardening.
**Risk:** Low.
**Dependencies:** None.

### Scope

- Support bracketed IPv6 tunnel hosts in `isValidRemoteHost()`.
- Example accepted: `[::1]`, `[2001:db8::1]`.
- Preserve rejection of unbracketed IPv6 (colons corrupt SSH `-L` format).
- Add bracket-aware parsing that extracts the host from `[addr]` for SSH argument construction.
- Add validation tests for bracketed IPv6 acceptance and unbracketed rejection.
- Do NOT weaken SSH `-L` argument safety — the `-L localPort:remoteHost:remotePort` format must remain correct.

### Design notes

- `RemoteTunnelRequest.isValidRemoteHost()` currently rejects `[` and `]` as shell metacharacters.
- To support `[::1]`, the validator needs a bracket-aware path: detect `^[...]$`, strip brackets, validate the inner address contains only hex digits, colons, and dots (IPv4-mapped).
- The `tunnelStart` argument builder must preserve brackets in the SSH `-L` argument (SSH itself accepts `[::1]:port`).

### Exit criteria

- `[::1]` accepted by `isValidRemoteHost()`.
- `::1` still rejected.
- `192.168.1.1` and `hermes.internal` still accepted.
- SSH `-L` argument builds correctly with bracketed host.
- Tests pass on CI.

## Phase 3: Live Remote Execution — Architecture Gate

**Size:** Large — planning only, no code until approved.
**Risk:** High (security-sensitive).
**Dependencies:** None (can start immediately).

### Questions this phase must answer

1. **What live remote operations are allowed?**
   - Read-only probes: `which hermes`, `hermes --version`, `hermes status`.
   - State queries: daemon status, tunnel status.
   - Destructive operations: `hermes restart` — should this ever be allowed in release builds?

2. **Which operations remain DEBUG-only?**
   - Any operation that modifies remote state.
   - Any operation that sends stdin data to the remote.
   - Tunnel start/stop.

3. **Which are allowed in release builds, if any?**
   - Minimum: `which hermes` and `hermes --version` (read-only, no side effects).
   - Conditional: `hermes status` (read-only but reveals internal state).
   - Gate: user must explicitly enable "Live Remote" mode in Settings with a confirmation dialog.

4. **What user confirmation is required?**
   - First-time live remote enablement requires explicit opt-in.
   - Each destructive command requires per-action confirmation.
   - Settings must display a clear "Live mode active" indicator.

5. **What logs are kept?**
   - All remote commands logged with `RemoteCommandInputMetadata` (command, byte count, timestamp).
   - Raw stdout/stderr never logged — only sanitised output.
   - Log entries include connection state, duration, and exit code.

6. **What outputs are redacted?**
   - All remote output passes through `OutputSanitiser` before display.
   - Credentialed URLs, API keys, bearer tokens, PEM keys, and secret assignments all redacted.
   - Streaming holdback prevents partial secret leakage.

7. **What rollback/kill switch exists?**
   - "Disable Live Remote" setting immediately reverts to mock mode.
   - No cached live connections survive app restart.
   - Kill switch does not require app restart — takes effect on next command.

8. **How does Solaris avoid becoming an arbitrary SSH frontend?**
   - Command allowlist is enforced at the enum level — `RemoteHermesCommand`.
   - No free-form shell access.
   - SSH invoked directly (`/usr/bin/ssh`) — never through a shell.
   - Custom `hermesCommandBase` is sanitised for shell metacharacters.
   - `BatchMode=yes` prevents interactive password prompting.
   - Stdin capped at 16KB.

### Safety contract (draft)

```
Solaris v0.11 Live Remote Safety Contract:

1. ALLOWED: Read-only probes (which, version, status) when user opts in.
2. BLOCKED: All destructive operations (restart, tunnel) in release builds.
3. BLOCKED: Arbitrary shell commands — only enumerated RemoteHermesCommand cases.
4. REQUIRED: User opt-in with confirmation dialog before first live command.
5. REQUIRED: All output sanitised through OutputSanitiser before display.
6. REQUIRED: Metadata-only logging — no raw payloads stored.
7. REQUIRED: Kill switch that immediately reverts to mock mode.
8. NEVER: Shell interpretation, password prompting, unbounded stdin, cached credentials.
```

### Implementation approach

- Define a `LiveRemotePolicy` enum with cases: `.disabled`, `.readOnly`, `.full`.
- Default: `.disabled`.
- Read-only allows: `whichHermes`, `hermesVersion`, `hermesStatus`.
- Full requires DEBUG build and is never available in release.
- `RemoteSSHExecutor` checks policy before executing.

### Exit criteria

- Architecture decision document approved.
- Safety contract reviewed and signed off.
- No code changes until approval.

## Phase 4: Remote Hermes Discovery Design

**Size:** Large — planning only, no code until Phase 3 architecture is approved.
**Risk:** Medium.
**Dependencies:** Phase 3 architecture decision.

### Questions this phase must answer

1. **Expected remote host assumptions?**
   - User has configured a remote host in Settings (hostname, username, port, SSH key).
   - SSH agent has the required key loaded.
   - Network path to the host is reachable.

2. **How does Solaris discover Hermes safely?**
   - Use the existing `whichHermes` command (already in the allowlist).
   - Use `hermesVersion` to confirm a compatible version.
   - Use `hermesStatus` to confirm it is running.
   - All three are read-only and produce no side effects.

3. **Discovery mechanism?**
   - Fixed commands (no dynamic discovery protocol).
   - Sequential probe: `which` → `version` → `status`.
   - Each step updates `RemoteConnectionState` and `RemoteHermesStatusSnapshot`.
   - No config file parsing, no API endpoint probing, no service mesh queries.

4. **How is failure displayed?**
   - `hermesFound: false` → "Hermes not found on remote host."
   - `hermesVersion: nil` → "Could not determine Hermes version."
   - SSH connection failure → sanitised error message via `OutputSanitiser`.
   - All failure states reflected in `RemoteConnectionState` enum.

5. **What is never read or logged?**
   - Raw SSH output is never logged directly.
   - Remote file system contents are never enumerated.
   - Remote environment variables are never queried.
   - SSH private key contents are never transmitted or logged.

6. **What happens when Hermes is not installed or not running?**
   - UI displays "Hermes unavailable" state.
   - Live remote controls are disabled.
   - User is directed to install Hermes on the remote host.
   - Mock mode remains available as fallback.

### Exit criteria

- Discovery protocol documented.
- Failure states mapped to UI.
- No implementation until Phase 3 is approved.

## Phase sequencing

```
Phase 1: IPv6 bracket support          (small, can start anytime)
Phase 2: CI workflow maintenance      (small, can start anytime)
Phase 3: Live execution architecture  (planning only, no code)
Phase 4: Remote Hermes discovery      (planning only, depends on Phase 3)
```

Phases 1 and 2 are independent and can run in parallel.
Phase 4 must not start implementation until Phase 3 is approved.

## Hard constraints

- Do not turn Solaris into an arbitrary SSH command runner.
- Do not weaken release-build safety gates.
- Do not combine live execution and remote discovery in the same batch.
- Do not log raw remote output.
- Do not cache SSH credentials or session state across app restarts.
- Do not enable destructive remote operations in release builds.
